# 模块完整介绍

为当前仓库生成可追溯、可验证、默认使用简体中文的模块介绍。让读者既能回答“模块做什么、为什么这样设计”，也能沿着真实代码说明“具体如何实现、每个参数从哪里来、运行时怎样生效、失败时怎样处理”。

## 核心约束

- 默认用简体中文撰写标题、正文、表格、图注和面试问答。仅在用户明确要求其他语言或仓库有强制语言规范时切换；代码标识符、配置键、SQL、类名和协议字段保留原文。
- 将“完整”定义为“已声明模块边界内的每个已发现实现面都有去向”：正文说明、附录登记、明确排除或标为未验证。不要把“扫描了若干代表文件”表述成完整覆盖。
- 只依据实际代码、测试、配置、迁移、ADR、计划和运行产物陈述事实。禁止凭经验补齐不存在的实现。
- 区分代码事实、运行产物、文档决策、推断和未验证项。禁止把计划、历史实验、取消工作或未执行检查写成已完成能力。
- 不泄露密钥、令牌、真实连接串、私有地址、用户数据、模型原始载荷或秘密配置值。秘密参数只写变量名、用途、来源层级和脱敏状态。
- 代码引用要精确到文件和符号；必要时引用短小的方法签名、枚举、不变量、SQL/Lua 片段。不要粘贴整类、整文件或大段受保护内容。
- 保留用户范围，不借模块介绍扩大成代码重构、架构改造或未经请求的实现工作。

## 强制工作流

### 1. 建立边界与交付路径

在编辑前：

1. 读取仓库的 `AGENTS.md`、`CLAUDE.md`、README、术语表、工程标准和相关 ADR；遵守其中的分支、文档位置、秘密处理和验证规则。
2. 运行 `git status --short --branch` 与 `git branch --show-current`，保留无关改动。
3. 写出模块边界：入口、出口、拥有的状态、直接依赖、直接消费者、明确不属于该模块的邻接能力。
4. 明确文档状态：当前实现、历史背景、计划能力、诊断能力、未验证能力分别是什么。
5. 告知用户预定文档与图片路径。若用户未指定，遵循仓库现有模式；无模式时使用 `docs/summary/<module-slug>.md` 和 `docs/summary/assets/<module-slug>-architecture.png`。

若模块过大，使用一个主文档和紧邻的中文附录拆分，例如参数、接口、数据模型或代码清单附录；主文档必须链接所有附录，附录仍属于同一完整交付。不要为了控制篇幅删掉已发现细节。

### 2. 生成证据清单

先用 `rg --files`、`rg` 和目录浏览定位全部相关面，再逐类阅读。至少盘点：

- 包、模块、入口、控制器、DTO、服务、领域对象、仓储、适配器、调度器、消费者、监听器、提示词、工具和运行器；
- 配置类、配置文件、环境变量文档、常量、注解默认值、启动参数、特性开关和第三方客户端设置；
- 数据库迁移、表、索引、唯一约束、ORM/Mapper、命名 SQL、Lua、缓存键和消息契约；
- 单元测试、集成测试、端到端测试、fixture、评测脚本、健康检查、日志/指标定义和实际运行产物；
- 计划、实现记录、ADR、既有模块文档及最近相关变更。

建立“覆盖台账”，每项至少记录 `类别 / 标识符 / 证据路径 / 正文位置 / 状态`。状态只能是：`已说明`、`明确排除`、`未验证`。发现同名或重复实现时，确认谁是权威实现，说明旧路径是否仍可达。

不得只因文件数量多而随机抽样。可以先读代表文件建立搜索词，再扩展到边界内所有命中项，直到覆盖台账没有未归类项。

### 3. 还原端到端架构和运行链路

按真实调用顺序追踪：

```text
入口/触发器 -> 协议转换 -> 应用编排 -> 核心规则/状态所有者
             -> 数据或基础设施适配 -> 外部依赖 -> 持久化/响应/事件
```

同时追踪正常路径、合法空结果、业务拒绝、校验失败、超时、取消、依赖失败、并发冲突、重试/降级和清理路径。说明每一跳增加的边界价值，不把无语义的转发层包装成设计亮点。

架构章节必须列出：

- 组件职责与所有权；
- 上游、下游和依赖方向；
- 控制流、数据流和状态流；
- 同步/异步边界、事务边界、线程或执行器边界；
- 外部系统、缓存、数据库、消息系统和模型/供应商边界；
- 关键设计取舍、已知替代方案及其证据。若源代码未记录选择原因，标为“推断”或“未记录”，不要编造。

### 4. 逐项追踪参数来源

对模块边界内所有影响行为的参数建立“参数来源与生效链路表”。包括但不限于：布尔开关、枚举、阈值、权重、超时、重试、批大小、并发度、队列容量、缓存 TTL、模型名、采样参数、token 限制、检索数量、分块尺寸、排序分数、调度周期、分页限制、端口、路径和安全策略。

每个参数至少记录：

| 字段 | 必须回答的问题 |
|---|---|
| 参数/外部键 | 代码字段、环境变量、YAML/JSON 键、CLI 参数或协议字段是什么？ |
| 语义与单位 | 它控制什么？单位、范围、枚举值或特殊值是什么？ |
| 声明位置 | 在哪个文件、类、构造器、注解、常量或 schema 中声明？ |
| 来源与优先级 | 来自配置文件、环境变量、系统属性、命令行、数据库、远程配置、请求、常量、运行时计算还是供应商默认？覆盖顺序是什么？ |
| 默认值来源 | 默认值由本项目显式给出、框架给出、供应商给出，还是没有默认值？证据在哪里？ |
| 绑定与转换 | 哪个 binder/factory/parser/mapper 读取它？是否有别名、单位换算、类型转换或空值处理？ |
| 校验与修正 | 是否必填？如何校验、clamp、归一化、拒绝或回退？ |
| 最终消费者 | 哪个方法、请求构造器、SQL、调度器或算法真正使用它？ |
| 行为影响 | 改大、改小、关闭或非法时会发生什么？ |
| 可观测性/保密 | 是否出现在日志、指标或管理接口？是否必须脱敏？ |

明确区分四类值：`声明默认值`、`部署配置值`、`本次运行有效值`、`运行时派生值`。只有实际读取运行环境后才能声称“当前有效值”；否则写默认值或配置来源，不得猜测。

对进入外部请求的参数，补充完整映射链：

```text
外部配置键/请求字段 -> 配置对象或 DTO -> 校验/换算 -> 内部选项
                        -> 供应商请求字段 -> 响应或运行行为
```

对于硬编码常量，也要说明设定位置和可找到的来源：测试契约、协议限制、供应商文档、历史 ADR、经验性选择或“仓库未记录依据”。不要把代码中的数字自动解释成行业最佳值。

### 5. 深挖特殊代码实现

对决定模块差异化行为、可靠性或性能的代码逐项解释，而不是只列类名。每个关键机制说明：

- 权威类/方法及其调用者；
- 输入、输出、前置条件和不变量；
- 分支顺序、算法步骤、状态变化和数据结构；
- 实际并发原语、锁、执行器、调度器和内存可见性保障；
- 事务开始/提交位置，事务外副作用和失败窗口；
- 重试、退避、超时、取消、熔断、降级、幂等、去重、补偿和清理；
- provider/格式专用解析、缓冲、提交/丢弃语义；
- 关键参数如何进入该机制；
- 对应测试如何覆盖成功、边界和失败分支。

优先用“代码深挖表”组织：`机制 / 权威符号 / 调用路径 / 核心步骤 / 状态或并发 / 参数 / 异常语义 / 测试证据`。若短代码比文字更清楚，只引用证明机制所需的最小片段并逐行解释。

对以下类型应用额外门槛：

- **RAG/解析**：按格式列解析器、库/引擎、输出片段、元数据、分块路线、视觉/OCR 路径、拒绝与回退；覆盖项目实际支持的 PDF、HTML、Markdown、Office、表格、图片等格式。
- **LLM 路由/流式**：列模型选择、提示词组装、工具调用、原始流解析、首包探测、缓冲提交/丢弃、取消、fallback 和熔断代际/陈旧结果保护。
- **异步/MQ/可靠性**：列队列、outbox、claim、锁所有权、watchdog、续租、重试、DLQ、恢复、幂等和 Redis/数据库失败策略；点名 SQL/Lua。
- **并发**：点名实际使用的 `CountDownLatch`、原子类、线程池、`ScheduledExecutorService`、行锁、Redis 锁、CAS 或其他原语及竞争结果。
- **数据契约**：列 schema、字段、索引、唯一约束、版本、幂等键、重复/陈旧/冲突处理和清理。
- **安全与隐私**：列认证、授权、租户/资源边界、秘密读取、日志脱敏、提示词/工具/provider 载荷边界。

源码没有某机制时明确写“未实现/未发现”，不要套用通用架构模板。

### 6. 生成并嵌入合适的图片

每个重要模块至少保留一个可维护的 Mermaid 图。根据关系选择：流水线用 `flowchart TD/LR`，多参与者时序用 `sequenceDiagram`，状态机用 `stateDiagram-v2`，数据关系用 `erDiagram`。

满足以下任一条件时，图片能明显降低理解成本，应调用 Codex 的 `$imagegen` 或可用 ImageGen 工具生成主架构图：

- 三个以上组件存在跨边界依赖；
- 同一入口分叉到多个 provider、处理路线或失败路线；
- 存在队列、缓存、数据库、外部系统共同参与的时序；
- 存在难以仅靠段落解释的状态机、回路或异步交接；
- 用户明确要求生成图片。

简单单类、单函数或一条线性链路不生成装饰图，Mermaid 足够时直接使用 Mermaid。

图片流程必须完整：

1. 先从代码证据画 Mermaid，确认组件、箭头、边界和标签。
2. 无需再次征求确认，调用 ImageGen；把架构事实、布局、短中文标签、边界、配色和禁止项写入 prompt。禁止把密钥、真实 URL、用户数据或未经验证的组件放进 prompt。
3. 使用白底、清晰分区、矢量感、有限蓝灰色、少量高亮；控制标签数量和长度，避免装饰图标、微小文字和不必要的 3D 效果。
4. 生成后进行视觉检查：文字是否可读、箭头是否正确、组件是否缺失、是否出现幻觉标签。存在问题就修订 prompt 并重新生成。
5. 将最终位图复制到仓库本地资源目录，使用稳定、语义化、无空格文件名。确认文件真实存在。
6. 在 Markdown 中使用相对路径嵌入图片，并在下方保留 Mermaid 作为可维护、可搜索的事实版本。
7. 写简短图注，说明图片路径、生成工具和“架构内容以代码与 Mermaid 为准”。

推荐 prompt 骨架：

```text
为 <仓库名> 的 <模块名> 生成一张中文技术架构图。
依据以下已验证组件与方向：<组件和箭头>。
突出边界：<边界>；侧边依赖：<依赖>；关键状态/分支：<状态或分支>。
风格：白色背景、现代工程文档、清晰二维分区、蓝灰配色、少量强调色、宽屏布局。
所有标签仅使用给定的短中文词和代码标识符；不要新增组件，不要出现密钥、真实配置值、装饰图标、3D 效果或微小文字。
```

若 ImageGen 未暴露、生成失败或无法取得可持久化文件，诚实记录原因，保留 Mermaid 和可复用的 `ImageGen prompt`。不得声称图片已生成，不得让仓库文档引用临时目录、会话 URL 或 `$CODEX_HOME/generated_images`。

### 7. 编写中文文档

除非用户指定其他结构，使用以下章节：

1. **标题、范围与状态**：受众、边界、当前/历史/计划状态、明确排除项。
2. **一句话定位与面试表达**：30 秒、2 分钟版本及需要记住的事实/数字。
3. **问题、目标与非目标**：模块为何存在，成功标准是什么。
4. **整体架构**：组件、职责、所有权、依赖方向、主架构图片与 Mermaid。
5. **设计与技术选型**：框架、库、模式、选择理由、代价和已记录的替代方案。
6. **端到端运行流程**：正常、空结果、拒绝、异常、重试、降级、取消和清理。
7. **接口与契约**：HTTP/事件/服务入口表，认证、DTO、字段、返回和错误语义。
8. **数据模型与持久化**：实体、表、索引、约束、迁移、Mapper/SQL、生命周期和幂等。
9. **参数来源与设定依据**：完整参数表、优先级、默认值依据、绑定、校验、消费者和行为影响。
10. **关键代码实现深挖**：类、方法、调用链、算法、状态机、并发、事务和短代码解释。
11. **可靠性、安全与可观测性**：超时、重试、熔断、锁、隐私、日志、指标和健康检查。
12. **验证与实际结果**：实际执行的命令、测试、产物、指标、失败诊断和 review 结果。
13. **限制、风险与未验证项**：不证明什么、剩余风险、缺失证据。
14. **扩展指南**：新增 parser/provider/consumer/route/rule 的落点和必须补充的测试。
15. **面试追问与回答**：围绕取舍、实现、故障和扩展给出简洁答案。
16. **覆盖台账与参考索引**：全部已验证路径、符号、测试、配置、迁移、产物和命令。

小模块可以合并相邻章节，但不得删掉适用信息。无持久化、无异步或无外部配置时，明确写“不适用”并给出搜索证据，不要静默省略。

## 证据标记

统一使用：

- **代码证据**：源文件、测试、配置、迁移或 schema 直接证明。
- **产物证据**：日志、报告、manifest、截图、指标或运行目录证明。
- **文档证据**：ADR、批准计划、实现记录或既有规范证明。
- **推断**：依据多处事实连接出的解释，明确标注推断依据。
- **未验证**：尚未运行、无法访问或证据不足。

所有数字、默认值、路径、类名、迁移名、SQL/Lua 名、配置键、测试名和图片路径都必须在交付前重新核对。计划文档只能证明意图，不能证明运行结果。

## 完整性终检

交付前逐项确认：

- 模块边界、权威行为和状态所有者已明确；上游、下游和依赖方向完整。
- 覆盖台账中没有无去向项；所有 `未验证` 均在限制章节可见。
- 接口、数据、配置、代码、并发、事务、异常、安全、观测、测试和扩展面都已处理或明确写“不适用”。
- 每个行为参数都有来源、默认依据、覆盖顺序、绑定/校验、消费者、单位和影响；秘密值没有泄露。
- 每个特殊机制都解释到方法、分支、状态、原语和失败语义，并连接测试证据。
- 所有引用路径和符号已用 `rg`、`Test-Path` 或等价命令验证存在；删除陈旧引用。
- 图片只在有信息价值时生成；生成图已视觉检查、复制到仓库、相对路径嵌入且保留 Mermaid。
- 未运行的测试、E2E、benchmark、live provider 或图片生成没有被写成已完成。
- 文档没有尾随空格、秘密、临时路径、失效链接或与用户无关的工作树改动。

至少运行：

```powershell
git diff --check -- <changed-files>
Select-String -Path <doc-and-appendices> -Pattern '[ \t]+$'
```

若文档被 Git 忽略，仍要运行尾随空格、路径存在性、秘密模式和陈旧声明检查。只有修改了代码、配置、prompt、schema 或脚本时才运行相应测试；纯文档修改不伪造构建需求。

## 技能衔接

- 模块尚未实现时，先使用 `$feature-implementer`，不要用文档代替实现。
- 需要按已批准计划或实现记录验收模块文档时，使用 `$cross-review`。
- review 返回具体文档 finding 时，使用 `$apply-review-fix` 做最小修复后重新 review。
- 用户实际要创建或修改可复用 Codex 技能时，改用 `$skill-creator`。
